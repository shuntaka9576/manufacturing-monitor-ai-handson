# 実装計画: 製造設備監視ダッシュボード データ基盤

## 概要

Excel ファイル（sample_data.xlsx）から SQLite データベースへシードデータを投入するスクリプトとスキーマを実装する。db/ ディレクトリにスキーマ定義とシードスクリプトを配置し、tests/ に検証テストを配置する。

## タスク

- [x] 1. プロジェクト構造とDBスキーマの作成
  - [x] 1.1 `db/` ディレクトリと `db/schema.sql` を作成する
    - 設計ドキュメントの DDL 定義に従い、equipment / status_history / sensor_data の3テーブルと2インデックスを定義する
    - _要件: 1.1, 1.5, 2.1, 2.2, 2.3, 2.4, 2.5, 2.6, 2.7_

- [x] 2. シードスクリプトのコア実装
  - [x] 2.1 `db/seed.py` にヘルパー関数を実装する
    - `to_sql_value`: NaN → None 変換ロジック
    - `validate_excel`: Excel ファイルの存在確認とシート名検証（設備マスタ / ステータス変更履歴 / センサーデータ）。検証失敗時は `SystemExit` を送出
    - `create_database`: 既存DB削除 → 新規作成 → schema.sql 読み込み → 外部キー制約有効化（`PRAGMA foreign_keys = ON`）
    - _要件: 1.2, 1.4, 3.5, 3.6, 3.7, 3.8, 3.9, 3.10, 3.13_

  - [x] 2.2 `db/seed.py` にデータ投入関数を実装する
    - `insert_equipment`: 設備マスタシートの全行を読み込み、行インデックス+1 を id として設定
    - `insert_status_history`: ステータス変更履歴シートの全行を読み込み投入
    - `insert_sensor_data`: センサーデータシートの全行を読み込み、NaN → NULL 変換して投入
    - _要件: 3.1, 3.2, 3.3, 3.4, 3.5, 3.13_

  - [x] 2.3 `db/seed.py` にトランザクション管理とメイン関数を実装する
    - `load_and_insert_data`: 単一トランザクション内で3テーブルへ投入。エラー時はロールバック
    - `print_summary`: 各テーブルのレコード件数と「シード完了」メッセージを標準出力に表示
    - `main`: エントリーポイント。`python db/seed.py` で実行可能にする
    - _要件: 3.11, 3.12, 4.1, 4.2, 4.3_

- [x] 3. チェックポイント - シードスクリプトの動作確認
  - `python db/seed.py` を実行し、DB ファイル生成とレコード件数表示を確認する。問題があればユーザーに確認する。

- [x] 4. 検証テストの実装
  - [x] 4.1 `tests/test_seed.py` にテストフィクスチャを作成する
    - シードスクリプトを実行して DB を生成するフィクスチャ（pytest）
    - Excel ファイルを読み込むフィクスチャ
    - _要件: 4.8_

  - [x]* 4.2 レコード件数と基本制約のテストを実装する
    - 各テーブルのレコード件数が期待値と一致することを確認（equipment: 8, status_history: 59, sensor_data: 1152）
    - 設備IDが1〜8の連番であることを確認
    - 外部キー制約が正しく機能していることを確認
    - CNC旋盤（設備ID 1,2）の rpm が NULL でないことを確認
    - _要件: 4.4, 4.5, 4.6, 4.7_

  - [x]* 4.3 ラウンドトリップ整合性テストを実装する
    - Excel の設備マスタ全行と equipment テーブル全行の一致を検証
    - Excel のステータス変更履歴全行と status_history テーブル全行の一致を検証
    - Excel のセンサーデータ全行と sensor_data テーブル全行の一致を検証（NaN は NULL として比較）
    - _要件: 5.1, 5.2, 5.3_

- [x] 5. 最終チェックポイント - 全テスト実行
  - 全テストが通ることを確認する。問題があればユーザーに確認する。

## 備考

- `*` 付きのタスクはオプションでスキップ可能
- 各タスクは対応する要件番号を参照しており、トレーサビリティを確保
- チェックポイントで段階的に動作を検証する
- プロパティテストはラウンドトリップ整合性テスト（タスク 4.3）がその役割を担う
